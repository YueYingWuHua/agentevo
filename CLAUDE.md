# CLAUDE.md — 技术演讲文档生成规则

> 本文件记录了本次「从Chat到Multi-Agent」技术分享的文档生成经验，
> 后续在其他项目生成演讲文档时可复用。

---

## 一、文档结构

### 1.1 演讲文档规范

每页 slide 以 `----` 分隔，包含三个固定段落：

```
### Slide N：标题

**核心要点**：
- 3-4 个 bullet points，听众带走的关键信息
- 写在这一页 slide 标题下方最醒目的位置

**演讲稿**：
> 口语化的演讲词，可逐句朗读
> 包含过渡钩子（连接上一页和下一页）
> 关键术语首次出现时做简要解释

**伪代码/图表**：
用代码块包裹，支持三种形式：
- Python-like 伪代码（简化版，去 import/异常处理/类型注解）
- ASCII 架构图
- 对比表格
```

### 1.2 技术文档 vs 演讲文档的职责分工

| | 技术文档 (`技术分享.md`) | 演讲文档 (`演讲文档.md`) |
|------|------|------|
| 定位 | 完整参考资料 | PPT 生成素材 |
| 内容 | 理论、原理、Demo 引用 | 演讲稿、伪代码、测试结果 |
| 代码 | 引用 Demo 路径 | 嵌入简版伪代码 |
| 详细程度 | 逐步展开 | 每页 3-4 个要点 |

**原则：技术文档是"教材"，演讲文档是"讲稿"。互不替代，互补。**

## 二、叙事规则

### 2.1 时间线优先

演讲必须按**技术演进的历史时间线**组织，不允许概念先行的跳转：

```
正确顺序: Chat → CoT → ReAct → Act进化 → Plan-and-Execute → Multi-Agent
错误顺序: Agent Loop 前置 → 再回头讲 Chat
```

### 2.2 并行研究要诚实标注

如果两个事件时间上有交叉或前后关系与直觉不符，必须在演讲稿和图表中明确标注：

```
例：CoT(2022.01) 和 ReAct(2022.10) 比 ChatGPT(2022.11) 更早。
    不能说"ChatGPT 之后业界开始研究推理"。
    应该说"三者最初是并行研究，2023 年后汇合"。
```

### 2.3 过渡钩子

每一节的最后一张 slide 末尾，必须有过渡语句连接下一节：

```
形式：
  "Act 可靠之后，新问题浮现 → Plan-and-Execute"
  "RAG 给了模型眼睛，但 Agent 需要手脚 → CoT + Tool Use"
```

章节之间的大过渡写在 `---` 分隔符之后。

## 三、代码与图表的呈现规则

### 3.1 伪代码标准

- 语言：Python-like，保留核心结构和注释
- 去掉：`import`、异常处理、类型注解
- 保留：函数签名、关键逻辑、内联注释
- 每条伪代码 ≤ 20 行

```python
# ✅ 好的伪代码
def agent_loop(task, max_steps=10):
    messages = [{"role": "system", "content": "你是Agent..."}]
    for step in range(max_steps):
        response = call_llm(messages)
        action = parse_action(response)   # 提取工具调用或 None
        if action is None:
            return response               # 任务完成
        result = execute_tool(action)
        messages.append({"role": "user", "content": result})
```

### 3.2 真实测试结果

- 直接嵌入已运行的测试输出，不依赖现场演示
- 标注是哪次运行的数据，方便追溯
- 对比场景（如 Prompt JSON vs 原生 FC）用左右对照

```
实际对比（同一任务，真实运行数据）:

📦 Prompt 强制 JSON:              🌐 原生 Function Calling:
  Step 1 ✅ JSON output             Step 1 ✅ tool_calls
  Step 2 ✅ JSON output             Step 2 ✅ tool_calls
  Step 3 ✅ JSON output             Step 3 ✅ tool_calls
  Step 4 ❌ 漂移！                  Step 4 ✅ tool_calls
  parse() 失败 → 退出               ...Step 8 ✅ 正常结束
```

### 3.3 对比表格式

对比两个概念时用 ASCII 表格：

```
        │ RAG                     │ Agent Tool Use               │
  模式   │ 检索 → 增强 Prompt → 回答  │ 思考 → 行动 → 观察 → 循环     │
  能力   │ 只读                     │ 读 + 写 + 执行 + 自我纠错     │
```

## 四、章节篇幅控制

### 4.1 每节 slide 数

| 复杂度 | 页数 | 示例 |
|--------|------|------|
| 简单过渡 | 1-2 页 | RAG 介绍 |
| 中等概念 | 2-3 页 | CoT、Plan-and-Execute |
| 核心机制 | 4-6 页 | ReAct/Agent Loop、Act 进化 |
| 总结 | 1 页 | 全篇总结 |

### 4.2 总时长估算

- 平均每页 slide 演讲耗时 2.5-4 分钟
- 伪代码/图表页偏短（听众看图，讲者补充），纯文字页偏长
- 60 分钟演讲 ≈ 18-22 页有效 slides（不含可选页）

## 五、内容红线

### 5.1 不准出现的内容

- "这里介绍一下"、"这里给出代码"等占位符——必须是实际内容
- 未来的预测当事实陈述——如"2025 年 Agent 会取代程序员"
- 学术术语不做解释直接使用——首次出现的专业名词必须用一句话解释
- 不诚实的时间线——如果两件事并行发生，不能编造成因果关系

### 5.2 必须出现的内容

- 每个阶段解决的核心问题（为什么需要它）
- 与上一阶段的衔接（它解决了上一阶段的什么痛点）
- 可运行的 Demo 引用或真实测试结果
- 关键论文的 arXiv ID 或年份

---

## 六、可选内容的处理

标记为 `[可选]` 的 slide：
- 在标题中标注 `[可选]`
- 演讲稿开头注明"视时间"
- 不依赖它作为后续章节的前提知识
- 全局时间估算中不计入

---

## 七、生成流程

1. **先确认需求**：受众、时长、深度、Demo 偏好（AskUserQuestion）
2. **产出大纲**：slide 标题列表 + 每页一句话描述，用户确认后再动笔
3. **逐节生成**：按时间线顺序，每完成一节让用户审阅
4. **全量完成后统一压缩**：根据总页数 × 3min 估算时长，超时则合并/删减
