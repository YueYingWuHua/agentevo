## 2. 核心概念：Agent Loop

在进入具体演进阶段之前，必须先理解 Agent 最核心的机制——**Agent Loop（Agent循环）**。

### 2.1 什么是 Agent Loop？

Agent Loop 是 Agent 系统的核心执行循环，其本质是一个**持续的 观察→思考→行动 循环**：

```
┌──────────────────────────────────────────────────┐
│                                                  │
│   ┌──────────┐    ┌──────────┐    ┌──────────┐  │
│   │ Observe  │───→│  Think   │───→│   Act    │  │
│   │ (观察)   │    │ (思考)   │    │ (行动)   │  │
│   └──────────┘    └──────────┘    └──────────┘  │
│        ↑                               │         │
│        └───────────────────────────────┘         │
│                                                  │
│   Agent Loop = while task_not_done:              │
│       observation = perceive(environment)        │
│       thought = reason(observation, history)     │
│       action = decide(thought)                   │
│       result = execute(action)                   │
│       // result 成为下一轮循环的 observation      │
└──────────────────────────────────────────────────┘
```

### 2.2 与纯 Chat 的本质区别

| 维度 | 纯 LLM Chat | Agent |
|------|------------|-------|
| 交互模式 | 一问一答，单轮/多轮对话 | 自主循环执行 |
| 状态管理 | 仅对话历史 | 对话历史 + 任务状态 + 环境反馈 |
| 工具使用 | 无 | 可调用外部工具/API |
| 终止条件 | 用户停止提问 | 任务完成 / 最大步数 / 用户中断 |
| 错误处理 | 用户重新提问 | 自主重试、调整策略 |

### 2.3 整个演进史，就是不断增强这个 Loop

| 演进阶段 | 增强了 Loop 的哪个环节 | 论文/事件 | 日期 |
|---------|----------------------|----------|------|
| **Chain of Thought** | **Think** — 让模型学会推理 | Wei et al., NeurIPS 2022 | 2022.01 |
| **ChatGPT 发布** | **Chat** — 多轮对话大众化 | OpenAI | 2022.11.30 |
| **ReAct** | **Think + Act + Observe** — 规范化完整循环 | Yao et al. | 2022.10 |
| **GPT-4 发布** | **能力基座** — 更强的推理和指令遵循 | OpenAI | 2023.03.14 |
| **硬编码 Tool Use** | **Act** — 让模型能调用工具（正则解析） | 工程实践 | 2022–2023 |
| **Function Calling** | **Act** — 结构化工具调用 | OpenAI | 2023.06.13 |
| **Plan-and-Solve** | **全局 Plan 层** — 先规划再求解 | Wang et al., ACL 2023 | 2023.05 |
| **MCP** | **Act** — 标准化 Agent-工具通信协议 | Anthropic | 2024.11.25 |
| **Agent Skills** | **Act** — 工具+知识+流程的复用单元 | Anthropic / 开放标准 | 2025.10.16 |
| **Multi-Agent** | **多 Loop 协作** — 多个 Agent 并行/串行 | 工程实践 | 2025–2026 |

## 3. 阶段一：纯LLM对话（ChatBox时代）

### 3.1 GPT-3 时代的工作场景（2020.06 — 2022.11）

在 ChatGPT 出现之前，LLM 的主流使用方式是 **Completion API + Few-shot Prompt**。没有"对话"概念，只有"文本补全"：

```
输入 Prompt:
  将以下文本分类为"正面"或"负面"：

  文本：这个产品非常好用 → 正面
  文本：物流太慢了，很失望 → 负面
  文本：客服态度很差，但产品质量不错 →

模型补全: 负面
```

这个时期 LLM 被当作一个**智能的 NLP 管道**来使用：

| 场景 | Prompt 方式 | 典型应用 |
|------|-----------|---------|
| **文本分类** | Few-shot 示例 + 待分类文本 | 客服工单分类、评论情感分析 |
| **信息提取** | 给出 Schema，让模型提取结构化字段 | 从简历提取姓名/学历/经验 |
| **摘要生成** | "Summarize the following article:" + 原文 | 新闻摘要、会议纪要 |
| **翻译** | Few-shot 翻译示例 + 待翻译文本 | 多语言内容本地化 |
| **内容生成** | 指定格式和风格，模型续写 | 营销文案、产品描述 |
| **代码补全** | 函数签名 + 注释，模型补全实现 | GitHub Copilot（2021.06 发布） |

核心特征：**Completion API，单次调用，无状态，每次请求独立**。

```
┌─────────────────────────────────────────────────┐
│  Prompt（开发者精心构造，包含指令+示例+输入）     │
│                        │                         │
│                        ▼                         │
│              ┌──────────────┐                    │
│              │  GPT-3       │                    │
│              │  (Completion │                    │
│              │   API)       │                    │
│              └──────────────┘                    │
│                        │                         │
│                        ▼                         │
│  Completion（模型续写的文本，取一次就结束）        │
└─────────────────────────────────────────────────┘
```

**根本局限**：
- 无法访问外部数据（不能查数据库、不能读文件、不能调 API）
- 无法执行连续的多步操作（一次调用 = 一次输入输出）
- 无法自我修正（输出错了不能自己发现和纠正）
- 上下文窗口极其有限（GPT-3 仅 2K tokens，`text-davinci-003` 也才 4K）

### 3.2 历史转折：ChatGPT（2022.11.30）

2022 年 11 月 30 日，OpenAI 发布 ChatGPT。这次变了什么？

| 维度 | GPT-3 时代 | ChatGPT |
|------|-----------|---------|
| **API 形态** | Completion API（文本补全） | Chat API（对话） |
| **交互模式** | 单次 Prompt → 单次补全 | 多轮对话，模型记住上文 |
| **对齐方式** | 纯 Prompt Engineering | RLHF（人类反馈强化学习） |
| **用户群体** | 开发者 | **全民**（5天百万用户） |
| **心智模型** | "这是一个 NLP 工具" | "这像一个能聊天的人" |

ChatGPT 是 Agent 发展史的**认知分水岭**——人们第一次觉得：**"如果它能对话，能不能让它帮我们做事？"**这个念头，就是 Agent 的起点。

### 3.3 关键突破：Multi-turn Conversation

ChatGPT 的核心技术突破之一是将历史对话作为 Context 持续传入：

```
User: "帮我写一个快速排序"
Assistant: "def quicksort(arr): ..."
User: "能加上类型注解吗？"
Assistant: "def quicksort(arr: List[int]) -> List[int]: ..."  ← 记住了上一轮的代码

实际实现：每轮请求把整个对话历史 messages 数组发给 API
messages = [
    {"role": "user", "content": "帮我写一个快速排序"},
    {"role": "assistant", "content": "def quicksort(arr): ..."},
    {"role": "user", "content": "能加上类型注解吗？"},
]
```

### 3.4 为什么纯 Chat 不够用？

ChatGPT 能"聊"了，但实际工作中很多任务它做不到：
- 读取本地文件 → LLM 无法访问文件系统
- 查询公司数据库 → LLM 无法连接数据库
- 调用第三方 API → LLM 无法发起网络请求
- 执行 Shell 命令 → LLM 无法操作终端
- 多步验证和自动纠错 → 纯对话模式效率极低

这催生了两个方向：**让模型学会推理（CoT）** 和 **让模型能调用工具（Tool Use）**。

> 📁 **代码示例**：`code/01_prompt_chat/simple_chat.py`

## 4. 阶段二：Chain of Thought（思维链）

### 4.1 问题：LLM 在复杂推理上表现不佳

直接让 LLM 回答复杂问题，准确率不高：

```
Q: "一个房间里有3个人，每人有2个苹果，后来进来了2个人，
    每人带了3个苹果，现在房间里一共有多少个苹果？"

直接回答: "12个" ← 可能算错
```

### 4.2 解决方案：Let's think step by step

**论文**：[Chain-of-Thought Prompting Elicits Reasoning in Large Language Models](https://arxiv.org/abs/2201.11903) — Wei et al., 2022.01

通过 Prompt 引导 LLM 展示推理过程，显著提高复杂任务的准确率。

**Zero-shot CoT**：只需在 Prompt 末尾加上 `"Let's think step by step"`：

```
Q: "一个房间里有3个人..."
A: Let's think step by step.
   第1步：初始3人，每人2个苹果 = 3 × 2 = 6个苹果
   第2步：后来2人，每人3个苹果 = 2 × 3 = 6个苹果
   第3步：总数 = 6 + 6 = 12个苹果
   答案是12个。
```

**Few-shot CoT**：给出几个带推理过程的示例，让模型学会推理模式。

### 4.3 CoT 的意义：Agent Think 环节的基础

CoT 证明了**LLM 可以"思考"**。这为后续所有 Agent 模式奠定了基础。

但 CoT 有个根本局限：**只能"想"，不能"做"**。它无法查询外部信息，无法执行操作。推理的正确性完全依赖模型内部知识，当模型不知道某个事实时，CoT 也无能为力。

> 📁 **代码示例**：`code/02_chain_of_thought/cot_demo.py`

## 5. 阶段三：ReAct模式

### 5.1 核心思想

**论文**：[ReAct: Synergizing Reasoning and Acting in Language Models](https://arxiv.org/abs/2210.03629) — Yao et al., arXiv 2022.10.06

ReAct（**Re**asoning + **Act**ing）的思路非常直接：
- CoT 让模型会思考 → 但只能"空想"
- 如果把"思考"和"行动"交替进行呢？每一步思考后执行一个动作，观察结果，再思考下一步...

核心范式：

```
Thought → Action → Observation → Thought → Action → Observation → ...
```

### 5.2 执行流程

```
用户的复杂任务
      │
      ▼
┌─────────────────────────────────────────────┐
│  Thought: "要完成这个任务，我需要先..."       │ ← 推理
├─────────────────────────────────────────────┤
│  Action:  search_database(query="xxx")       │ ← 行动
├─────────────────────────────────────────────┤
│  Observation: "查询返回了 5 条结果..."        │ ← 观察
├─────────────────────────────────────────────┤
│  Thought: "这些结果中第3条最相关，接下来..."   │ ← 再推理
├─────────────────────────────────────────────┤
│  Action:  read_file(path="/data/report.md")  │ ← 再行动
├─────────────────────────────────────────────┤
│  Observation: "文件内容显示..."               │ ← 再观察
├─────────────────────────────────────────────┤
│  Thought: "现在信息足够了，可以给出答案"       │ ← 判断终止
├─────────────────────────────────────────────┤
│  Final Answer: "根据分析，结论是..."          │ ← 输出结果
└─────────────────────────────────────────────┘
```

### 5.3 ReAct vs 纯 CoT vs 纯 Tool Use

| 模式 | 能思考 | 能行动 | 能纠错 |
|------|--------|--------|--------|
| 纯 CoT | ✓ | ✗ | ✗ |
| 纯 Tool Use | ✗ | ✓ | ✗ |
| **ReAct** | ✓ | ✓ | ✓ |

ReAct 是第一个将三者统一的范式。

### 5.4 为什么 ReAct 是 Agent 的基石？

- **交替推理与行动**：不是"想好了再做"，而是"边想边做，边做边调整"
- **基于观察的决策**：每一步决策都基于上一步的真实结果，而非想象
- **自我纠错**：如果某一步工具调用失败，Agent 能观察到错误并调整策略
- **可解释性**：Thought 步骤让 Agent 的决策过程透明可审查

### 5.5 ReAct 的第一个实现：硬编码 Tool Use

ReAct 论文（2022.10）不仅是理论范式——它有一个**可运行的原型实现**。这个实现就是硬编码工具调用：

```
论文中 HotpotQA 任务的真实执行流程：

  Thought: "I need to search for the director of Inception"
  Action: Search[Inception director]         ← LLM 输出
           │
           ▼ 正则提取 Search[...] 的内容
           调用 Wikipedia API
           │
           ▼
  Observation: "Inception was directed by   ← API 返回结果
               Christopher Nolan..."
  Thought: "The answer is Christopher Nolan" ← LLM 基于 Observation 推理
  Final Answer: Christopher Nolan
```

**关键事实**：ReAct 的 Observation 来自工具执行结果。没有工具调用，就没有 Observation；没有 Observation，ReAct 循环就转不起来。**硬编码 Tool Use 不是独立于 ReAct 的东西——它就是 ReAct 的第一个实现方式。**

但当时的硬编码方案非常原始：

- ChatGPT 还没发布（ReAct arXiv 提交于 2022.10.06，ChatGPT 发布于 2022.11.30），GPT-4 更是一年之后的事
- 没有 Function Calling API，LLM 输出的 Action 靠正则解析
- 没有标准化的工具协议，每个工具单独写解析逻辑
- Action 格式稍有偏差就解析失败，**整个 ReAct 循环就断了**

ReAct 给出了 Agent 的"骨架"（Think→Act→Observe 循环），但"肌肉"（Act 环节的可靠性）完全跟不上。

**接下来从 2023 到 2025 年，整个工具调用演进史，本质上就是在不断强化 ReAct 的 Act 环节——从脆弱的正则解析，到 Function Calling API（2023.06）的结构化输出，到 MCP 协议（2024.11）的标准化通信，最终到 Agent Skills（2025.10）的能力封装复用。**

> 📁 **代码示例**：`code/04_react_loop/react_agent.py`

## 6. 阶段四：Act 环节的进化——从硬编码到 Skill

> ReAct 论文(2022.10)确定了 Think→Act→Observe 的循环模式，其中 Act 环节最初用硬编码实现。从那时起到 Agent Skills(2025.10)，近三年时间，业界在解决同一个问题：**如何让 ReAct 的 Act 环节从"脆弱的正则解析"进化为"工业级可靠"**。

### 6.1 硬编码工具调用（ReAct 的原始 Act 实现）

#### 6.1.1 原理

硬编码工具调用是 ReAct 论文的**原始实现方式**，也是 2022-2023 年间开发者实现 Agent 工具调用的唯一手段：

```
1. 在 Prompt 中描述可用工具
2. LLM 输出特定格式的 Action 文本（如 Search[...] 或 {"tool": "xxx"}）
3. 用正则表达式或字符串解析提取工具名和参数
4. 执行对应的函数（调 API、读文件等），结果即为 Observation
5. 将 Observation 拼回 Prompt，LLM 据此生成下一个 Thought
```

这就是完整的 **ReAct 一步循环**：

```
┌─────────────┐     ┌──────────────┐     ┌──────────────┐
│ Thought:    │────→│ Action:      │────→│ 正则解析      │
│ "需要搜索X" │     │ Search[X]    │     │ 提取参数 → X  │
└─────────────┘     └──────────────┘     └──────┬───────┘
       ↑                                        │
       │         ┌──────────────┐                │
       └─────────│ Observation  │←───────────────┘
                 │ "结果是..."  │   执行 Wikipedia API
                 └──────────────┘
```

#### 6.1.2 痛点

- **脆弱的解析**：LLM 输出格式稍有变化就解析失败
- **难以扩展**：每加一个工具都要修改正则/解析逻辑
- **无类型安全**：参数类型完全靠字符串传递
- **上下文浪费**：工具描述占据大量 Token

> 📁 **代码示例**：`code/03_hardcoded_tool/hardcoded_tool.py`

### 6.2 Function Calling

#### 6.2.1 原理（2023.06，OpenAI）

OpenAI 推出 Function Calling API，将工具调用**结构化**：

- **开发者**：定义 Function Schema（函数名、描述、参数 JSON Schema）
- **LLM**：根据用户问题，选择是否调用工具，输出结构化的 Function Call
- **开发者**：执行函数，将结果返回给 LLM
- **LLM**：基于结果生成最终回复

```
┌─────────────┐     ┌──────────────┐     ┌──────────────┐
│ Function    │────→│ LLM 判断      │────→│ 结构化输出    │
│ Schema      │     │ 是否需要工具   │     │ {name, args} │
└─────────────┘     └──────────────┘     └──────────────┘
                                                │
                    ┌──────────────┐             │
                    │ LLM 基于结果  │←────────────┤
                    │ 生成最终回复   │  执行函数     │
                    └──────────────┘             │
```

#### 6.2.2 为什么是里程碑？

- **结构化**：不再依赖脆弱的字符串解析
- **类型安全**：基于 JSON Schema 的参数定义
- **多工具选择**：LLM 自动判断该调用哪个工具
- **可组合**：一次可调用多个工具

> **注**：Function Calling 是后续 MCP 和 Skill 的底层基础。这里不单独提供代码 Demo，因为后续 MCP 和 Skill 的 Demo 本质上都是基于 Function Calling 模式的。

### 6.3 MCP（Model Context Protocol）

#### 6.3.1 问题：Function Calling 的局限

Function Calling 解决了格式问题，但带来了新的挑战：

- **工具发现**：每个工具需要手动在代码中注册，Agent 切换环境时需重新配置
- **工具标准化**：不同服务的工具定义格式不统一
- **通信协议**：工具执行在同一个进程中，无法跨进程/跨网络调用

#### 6.3.2 MCP 是什么？（2024.11.25，Anthropic 发布）

MCP（Model Context Protocol）是一种**标准化的 Agent-工具通信协议**：

```
┌──────────┐    JSON-RPC over stdio/HTTP    ┌──────────────┐
│  MCP     │←─────────────────────────────→│  MCP          │
│  Client  │   tools/list, tools/call,      │  Server       │
│  (Agent) │   resources/read, prompts/get  │  (Tool Host)  │
└──────────┘                                └──────────────┘
```

**核心设计**：
- **Client-Server 架构**：Agent（Client）通过标准协议连接工具服务（Server）
- **传输层无关**：支持 stdio（本地进程）、HTTP/SSE（远程服务）
- **能力发现**：Client 可以动态查询 Server 提供哪些工具
- **结构化交互**：基于 JSON-RPC 2.0

#### 6.3.3 MCP 的核心概念

| 概念 | 说明 | 示例 |
|------|------|------|
| **Tools** | Server 提供的可调用工具 | `list_files`, `search_code`, `query_db` |
| **Resources** | Server 暴露的数据资源 | 文件内容、数据库表、API 响应 |
| **Prompts** | Server 预定义的 Prompt 模板 | 代码审查模板、报告生成模板 |

#### 6.3.4 MCP 解决了什么？

| 问题 | MCP 的解决方式 |
|------|---------------|
| 工具发现 | Client 调用 `tools/list` 动态发现，无需硬编码 |
| 跨进程调用 | 基于 stdio/HTTP，工具可运行在独立进程 |
| 工具标准化 | 统一的 JSON-RPC 协议，Server 可复用 |
| 权限隔离 | Server 独立进程，崩溃不影响 Agent |

> 📁 **代码示例**：`code/05_mcp_demo/`（本地 MCP Server + Client，纯本地运行）

### 6.4 Skill 模式

#### 6.4.1 什么是 Skill？（2025.10.16，Anthropic 发布，同年 12 月成为开放标准）

Skill 是一种**更高层次的工具封装模式**。它将一组相关的工具、知识和 Prompt 打包为一个可复用的"技能"。

```
Skill = Prompt Template + Tools + Domain Knowledge + Workflow
```

```
┌─────────────────────────────────────────┐
│ Skill: "CodeReview"                     │
│                                         │
│  ├─ Prompt: "请审查以下代码..."          │
│  ├─ Tools: [read_file, run_linter, ...] │
│  ├─ Knowledge: 编码规范、常见漏洞模式     │
│  └─ Workflow: 读取 → 检查 → 报告        │
└─────────────────────────────────────────┘
```

#### 6.4.2 Skill vs 硬编码工具

| 维度 | 硬编码工具 | Skill模式 |
|------|-----------|----------|
| **维护成本** | 每个工具独立维护，工具间耦合 | Skill内聚，修改在Skill内部 |
| **复用性** | 工具级复用，需要手动组合 | Skill级复用，开箱即用 |
| **上下文占用** | 每个工具的Schema都需注入Prompt | Skill按需加载，减少Token浪费 |
| **知识传递** | 仅提供工具描述 | 包含领域知识、最佳实践 |
| **可组合性** | 需要手动编排调用顺序 | Skill内置Workflow，自动编排 |
| **版本管理** | 散落在代码各处 | Skill独立版本化 |

#### 6.4.3 当前生态

Skill 模式仍在快速发展中。参考：[agentskills.io](https://agentskills.io/home)

> 📁 **代码示例**：`code/06_skill_demo/skill_agent.py`

```
ReAct 论文原始实现      Function Calling         MCP                    Agent Skills
硬编码 Tool Use
(2022.10)              (2023.06.13)             (2024.11.25)           (2025.10.16)
     │                      │                       │                      │
     ▼                      ▼                       ▼                      ▼
```

## 7. 阶段五：Plan-and-Execute

### 7.1 问题：ReAct 的局限

工具调用能力成熟后，ReAct 模式暴露出一个根本问题——**缺乏全局视野**：

- Agent 专注于当前这一步的推理和行动
- 没有对整个任务的全局规划
- 容易出现"走一步看一步"导致的路径偏离
- 复杂任务可能在中途迷失方向

### 7.2 解决方案：两阶段架构

**论文**：[Plan-and-Solve Prompting](https://arxiv.org/abs/2305.04091) — Wang et al., ACL 2023 (arXiv: 2023.05)

```
                    用户任务
                       │
           ┌───────────┴───────────┐
           ▼                       ▼
     ┌──────────┐           ┌──────────┐
     │ Planner  │──────────→│ Executor │
     │ (规划器) │  生成Plan  │ (执行器) │
     └──────────┘           └──────────┘
                                 │
                                 │ 按步执行
                                 ▼
                           ┌──────────┐
                           │ ReAct    │
                           │ Loop     │
                           │ (每步用  │
                           │  ReAct   │
                           │  执行)    │
                           └──────────┘
```

**Planner**：将复杂任务分解为有序的子任务列表
**Executor**：使用 ReAct 模式逐步执行每个子任务，将结果反馈给 Planner

### 7.3 与现代 Agent 的关系

现代 Agent 系统往往是 **Plan-and-Execute + ReAct** 的混合架构：

- **Planner** 负责制定和调整计划
- **Executor** 使用 ReAct Loop 执行每个步骤
- **动态重规划**：Executor 的执行结果可能触发 Planner 调整计划

> 📁 **代码示例**：`code/07_plan_execute/plan_execute_agent.py`

## 8. 阶段六：Multi-Agent与前沿进展

### 8.1 Multi-Agent 协作

#### 8.1.1 为什么需要 Multi-Agent？

单个 Agent 的能力有上限，复杂任务需要多个专业 Agent 协作：

- **角色分工**：不同 Agent 负责不同领域（如 Coder Agent、Reviewer Agent、Tester Agent）
- **并行处理**：独立子任务可以并行执行
- **交叉验证**：多个 Agent 相互检查，减少幻觉

#### 8.1.2 主要协作模式

| 模式 | 描述 | 适用场景 |
|------|------|---------|
| **Sequential** | Agent 按顺序执行，前一个的输出是后一个的输入 | 流水线任务 |
| **Hierarchical** | 一个 Master Agent 调度多个 Worker Agent | 复杂项目管理 |
| **Debate** | 多个 Agent 讨论并达成共识 | 需要多角度分析的任务 |
| **Parallel + Merge** | 多 Agent 并行处理，结果汇总 | 数据分析、批量处理 |

```
Sequential:
  [Agent A] → [Agent B] → [Agent C] → Result

Hierarchical:
       [Master Agent]
        /    |    \
  [Agent A][Agent B][Agent C]
        \    |    /
       [Merge Result]

Debate:
  [Agent A] ←→ [Agent B]
       ↘      ↙
     [Consensus]
```

> 📁 **代码示例**：`code/08_multi_agent/multi_agent.py`

### 8.2 Computer Use

#### 8.2.1 概念

Computer Use 让 Agent 能够像人类一样操作计算机界面：
- 查看屏幕截图
- 移动鼠标、点击
- 键盘输入
- 操作 GUI 应用程序

#### 8.2.2 意义

将 Agent 的能力从"调用 API"扩展到"操作任何软件"——即使没有 API 的传统软件也能被 Agent 操控。

### 8.3 Code Agent

#### 8.3.1 概念

Code Agent 是一种专门用于软件开发的 Agent：
- 读写代码文件
- 执行 Shell 命令
- 运行测试、检查 Lint
- 提交代码、创建 PR
- 自主 Debug 和修复

#### 8.3.2 当前状态

Code Agent 是目前最成熟的 Agent 应用领域之一（如 Claude Code、Cursor、Copilot 等），其核心是：
- Agent Loop + 文件系统工具 + 终端工具 + 代码智能

> 📁 **代码示例**：`code/08_multi_agent/computer_use_demo.py`（可选）

## 9. [可选] 进阶话题：上下文压缩与渐进式披露

> ⚠️ **以下内容为可选章节，视时间决定是否讲解。**

### 9.1 上下文压缩（Context Compression）

#### 9.1.1 问题

Agent 在 Loop 执行过程中上下文不断增长：
- 每一轮都追加 Thought、Action、Observation
- 很快超出 LLM 的 Context Window
- 即使不超出，长上下文也会导致推理速度变慢、注意力稀释（Lost in the Middle）、Token 成本增加

#### 9.1.2 主流方案

| 方案 | 原理 | 优缺点 |
|------|------|--------|
| **滑动窗口** | 只保留最近 N 轮对话 | 简单但丢失早期关键信息 |
| **摘要压缩** | 用 LLM 将历史对话压缩为摘要 | 保留关键信息但可能遗漏细节 |
| **结构化记忆** | 将信息分类存储（长期/短期/工作记忆） | 灵活但实现复杂 |
| **向量检索** | 将历史转为 Embedding，按需检索 | 精准但依赖 Embedding 质量 |

### 9.2 渐进式披露（Progressive Disclosure）

#### 9.2.1 问题

Agent 可能拥有几十甚至上百个工具。如果把所有工具的 Schema 都注入 System Prompt：
- 大量 Token 浪费在工具描述上
- LLM 的注意力被稀释
- 工具选择准确率下降

#### 9.2.2 解决方案

不在初始 Prompt 中列出所有工具，而是按需暴露：

```
初始 Prompt：只包含工具分类摘要
     │
     │ LLM 判断需要"文件操作"类工具
     ▼
注入文件工具 Schema：[read_file, write_file, list_files]
     │
     │ LLM 判断还需要"代码搜索"
     ▼
注入搜索工具 Schema：[search_code, grep]
     │
     ...
```

**实现方式**：
- **工具分组**：将工具按功能分类
- **按需加载**：LLM 表示需要某类工具时再注入详细 Schema
- **动态卸载**：不常用的工具 Schema 从上下文中移除
