## 一、Chat 时代：LLM 怎么从工具变成对话伙伴（3-4 页）

----

### Slide 1：GPT-3 时代的 LLM —— Completion API

**核心要点**：
- ChatGPT 之前，LLM 的主流用法是 Completion API
- 没有"对话"概念，只有"文本补全"
- GPT-3（2020.06）to ChatGPT（2022.11），两年半的沉默期

**演讲稿**：
> 在座各位应该都用过 ChatGPT，但 ChatGPT 之前 LLM 长什么样？
>
> 2020 年 6 月 GPT-3 就发布了，但那时候没有"聊天"这回事。
> 你给 API 发一段文本，模型帮你"续写"一段文本。用完就结束，下一次调用不记得你。
>
> 当时的主流场景是什么？情感分析、文本分类、摘要生成、翻译——
> 全是 NLP 流水线上的一个环节。开发者通过 Few-shot Prompt 来"教"模型怎么做。
> 比如给 3 个示例，"正面/负面/正面"，模型照着模式输出。
>
> 你看，这离我们今天聊的 Agent 还很远。但 2022 年 11 月，一切都变了。

**伪代码/图表**：
```
GPT-3 Completion API（2020-2022）:

输入（Prompt）:
  将以下文本分类为"正面"或"负面"：

  文本：这个产品非常好用 → 正面
  文本：物流太慢了，很失望 → 负面
  文本：客服态度一般，但产品不错 →

输出（Completion）: 正面

特点：单次调用，无状态，没有"对话历史"这个概念。
```

### Slide 2：ChatGPT —— 对话革命（2022.11.30）

**核心要点**：
- ChatGPT 把 API 从 Completion 变成了 Chat
- 多轮对话 + 上下文记忆 + RLHF 对齐
- 5 天百万用户，认知的转变："它像个能聊天的人"

**演讲稿**：
> 2022 年 11 月 30 号，ChatGPT 发布。从技术上看它做了什么？
>
> 第一，把 Completion API 改成了 Chat API。
> Completion 每次请求是独立的，Chat 把整个对话历史 messages 数组传给模型。
> 所以模型"记得"你刚才说了什么。
>
> 第二，RLHF(Reinforcement Learning from Human Feedback) 对齐。通过人类反馈强化学习，让回复更符合人类期望。
> 以前 GPT-3 的直接回复可能很生硬，ChatGPT 的回复让人觉得"它在跟我聊天"。
>
> 第三，用户体验的质变。5 天百万用户，大家第一次觉得
> "AI 不是一个工具，它像一个可以对话的伙伴"。
>
> 这个认知的转变，就是 Agent 的起点——
> 因为一旦你觉得它能对话，下一个念头就是："那它能帮我做事吗？"

**伪代码/图表**：
```
GPT-3 Completion API:                ChatGPT Chat API:
  每次新建连接                         维护 messages 数组
  无对话历史                           有对话历史
  开发者使用                           全民使用

ChatGPT 的多轮对话实现:
  messages = [
    {"role": "system", "content": "你是助手"},
    {"role": "user", "content": "帮我写快速排序"},
    {"role": "assistant", "content": "def quicksort(arr): ..."},
    {"role": "user", "content": "能加上类型注解吗？"},       ← 模型"记住"了上一轮的代码
  ]
  每次请求把整个 messages 发给 API
```

### Slide 3：为什么 ChatGPT 不够——它看不到你的数据

**核心要点**：
- ChatGPT 的知识来自训练数据，截止日期固定
- 不能读你的本地文件、不能查你的数据库、不能访问你的邮件
- 这催生了两个方向：学会推理（CoT）和学会动手（Tool Use）

**演讲稿**：
> 好，ChatGPT 会聊天了。但很快大家发现了一个根本问题——
>
> ChatGPT 很聪明，但它是一个"被困在训练数据里"的聪明。
> 你问它"公司内部最新的产品定价是多少"——它不知道，因为定价文档没在训练数据里。
> 你问它"上周三客户发来的邮件里提到了什么需求"——它看不到你的邮箱。
> 你问它"帮我确认一下这份 40 人的名单里谁不在公司通讯录里"——它无法查询你们的数据库。
>
> 注意，这里的需求不是"帮我写代码"。事实上 2022 年底 ChatGPT 的代码能力还很有限——
> 主要是代码补全和简单函数，让它写一个完整的项目代码经常出错。
> 真正驱动 Agent 发展的，是更朴素的需求：**"模型很聪明，但看不到我的数据"**。
>
> 它就像一个非常博学但被反锁在房间里的人——有知识，但没有感官。
> 他不知道这个房间外面在发生什么，也无法去查看任何外部的信息。
>
> 所以从 2022 年底开始，业界在两个方向上突破：
> 第一，让模型学会"思考"——复杂问题分步推理；
> 第二，让模型学会"动手"——调用外部工具、访问外部数据。
>
> 这两个方向最终在 ReAct 中汇合了。

**伪代码/图表**：
```
ChatGPT 能做:                       ChatGPT 做不了:
  回答训练数据覆盖的问题              查询你公司的内部数据库
  生成简单代码片段（补全/函数）        读取你的本地文件
  翻译文本                           访问你的邮箱或日历
  解释公开知识概念                    执行外部 API 调用

驱动 Agent 的真实需求（2022 年底 ~ 2023 年初）：
  "帮我查一下这份 PDF 合同里的违约条款"
  "帮我在公司通讯录里确认这几个人的联系方式"
  "帮我看看最近一周收到的邮件里有没有提到这个项目延期"
  "帮我对比客户发来的三份报价单，找出最优方案"

两个突破方向:
  方向一: 让模型学会"推理" → Chain of Thought (2022.01)
  方向二: 让模型学会"动手" → Tool Use → ReAct (2022.10)

  注意：CoT（2022.01）和 ReAct（2022.10）比 ChatGPT（2022.11）更早提出。
  它们最初是并行研究——ChatGPT 解决"对话界面"问题，
  CoT/ReAct 解决"推理与行动"问题。2023 年后才交汇融合。
  以下按概念递进而非严格时间顺序讲述。
```

### Slide 4：RAG——第一次尝试让模型"看见"外部数据

**核心要点**：
- RAG = 先检索再回答，把外部数据作为上下文注入 Prompt
- 原理：用户提问 → 在文档库中检索相关内容 → 拼入 Prompt → LLM 基于检索结果回答
- RAG 让模型能"读"，但它不能让模型"做"
- 优势：简单有效，劣势：只读，无法执行操作

**演讲稿**：
> Slide 3 我们说了 ChatGPT 最大的痛点是看不到你的数据。
> 学术界对这个问题的第一个系统性回答，叫 RAG——Retrieval-Augmented Generation，检索增强生成。
>
> 原理非常朴素：既然模型没见过你的文档，那在提问之前，先帮你"查"一下。
>
> 用户问"合同里的违约金是多少"——先在你的 PDF 库里用语义搜索找到相关段落，
> 把这些段落的原文拼到 Prompt 里："根据以下文档内容回答：...[检索到的段落]... 问题：违约金是多少？"
> 然后 LLM 基于这些拼进去的内容来回答。
>
> RAG 在 2023 年非常流行，几乎所有企业内部知识库问答系统都是这个架构。
> 它确实解决了一部分问题——模型现在知道你的数据了。
>
> 但 RAG 有一个致命限制：它**只能读**。模型可以基于检索到的内容回答你，
> 但如果你的需求是——"根据合同内容，帮我在系统里把下个月的付款金额改成 5000"，
> RAG 做不了，因为它只有检索+回答，没有执行能力。
>
> RAG 给了模型"眼睛"，但 Agent 需要给模型"手脚"。

**伪代码/图表**：
```
RAG 架构:

  用户提问: "违约金是多少？"
        │
        ▼
  ┌─────────────────┐
  │ 1. Embedding    │  把问题转为向量
  └────────┬────────┘
           ▼
  ┌─────────────────┐
  │ 2. 向量检索      │  在文档库中找到最相关的段落
  │ 找到: "若逾期支付，│
  │ 每日违约金0.05%"  │
  └────────┬────────┘
           ▼
  ┌─────────────────┐
  │ 3. 拼入 Prompt   │  "根据以下文档片段回答用户问题：
  │                 │   文档：若逾期支付，每日违约金0.05%
  │                 │   问题：违约金是多少？"
  └────────┬────────┘
           ▼
  ┌─────────────────┐
  │ 4. LLM 生成回答  │  "违约金为每日 0.05%。"
  └─────────────────┘
```

```
RAG vs Agent Tool Use:

        │ RAG                     │ Agent Tool Use               │
  模式   │ 检索 → 增强 Prompt → 回答  │ 思考 → 行动 → 观察 → 循环     │
  能力   │ 只读                     │ 读 + 写 + 执行 + 自我纠错     │
  范围   │ 文档/知识检索             │ 文件系统 / 数据库 / API / CLI │
  代表   │ 企业知识库问答             │ Agent（本分享的主角）         │

RAG 的遗产: "把外部数据注入 Prompt" 这个思路，
           后来成为 Agent Memory 中从长期记忆检索信息的核心机制。
```

## 二、Chain of Thought：让 LLM 学会"思考"（并行于 ChatGPT 的独立研究线，2-3 页）

----

### Slide 5：LLM 在复杂推理上表现不佳

**核心要点**：
- 直接回答复杂推理问题，LLM 准确率不高
- 不是知识不够，是"懒得想"
- CoT 的思路：引导 LLM 把推理过程写出来

**演讲稿**：
> 刚才 Slide 3 的时间线注释提到，CoT 和 ReAct 与 ChatGPT 是并行的独立研究。
> 我们先看 CoT——它在 2022 年 1 月就发表了，比 ChatGPT 早了近一年。
> 但它解决的问题至今仍是 Agent 推理能力的基础。
>
> 一个简单的例子。问模型"房间 3 个人每人 2 个苹果，后来进 2 人每人 3 个苹果，共多少个苹果"——
> 这种题小学生都能做，但 LLM 直接回答时有一定错误率。
>
> 不是因为模型不知道 3×2=6，而是它"懒得想"——直接跳到答案。
> 就像考试时你要求写计算过程，学生才会一步步算。
>
> 这就是 2022 年 1 月 Chain of Thought 论文的核心思想：
> 在 Prompt 里引导模型"一步步推理"，让它把中间过程写出来。

**伪代码/图表**：
```
问题: "房间 3 人，每人 2 个苹果；又进 2 人，每人 3 个苹果。共多少个苹果？"

❌ 直接提问:
  输入: "请回答：有多少个苹果？"
  输出: "12 个"     ← 可能算错

✅ Zero-shot CoT:
  输入: "有多少个苹果？Let's think step by step."
  输出: "第1步：初始苹果数 = 3×2 = 6
         第2步：新增苹果数 = 2×3 = 6
         第3步：总数 = 6+6 = 12
         答案是 12 个。"     ← 准确率显著提高
```

----


### Slide 6：Zero-shot CoT 和 Few-shot CoT

**核心要点**：
- Zero-shot CoT：加一句 "Let's think step by step"
- Few-shot CoT：给几个带推理过程的示例

**演讲稿**：
> CoT 有两个版本。
>
> Zero-shot CoT 最简单——在 Prompt 末尾加一句 "Let's think step by step" 就够了。
> 这句话就像一个开关，告诉模型"你别急着给答案，一步一步来"。
>
> Few-shot CoT 更精确——你在 Prompt 里给 2-3 个带完整推理过程的示例，
> 模型照着示例的格式输出。它的准确率比 Zero-shot 更高，但需要精心设计示例。
>
> CoT 的意义超越了"提高准确率"——
> 它第一次证明了 LLM 的输出不是只能"直接给答案"，而是可以展示"推理过程"。
> 这个"展示过程"的能力，后来成了 Agent Think 环节的基石。

**伪代码/图表**：
```
Zero-shot CoT:
  输入: "{问题}\nLet's think step by step."
  特点: 一句话就够了，零样本

Few-shot CoT:
  输入: "示例1: {问题1} → 推理: {过程1} → 答案: {答案1}
         示例2: {问题2} → 推理: {过程2} → 答案: {答案2}
         现在回答: {问题3}"
  特点: 给出推理范例，模型模仿格式

CoT 的遗产:
  证明 LLM 可以展示"推理过程" → 成为 Agent Think 环节的基础
```

----

### Slide 7：CoT 的意义和根本局限

**核心要点**：
- CoT 让模型能"想"，但不能"做"
- 推理的正确性完全依赖模型内部知识
- 当模型不知道某个事实时，CoT 无能为力 → 需要 Tool Use

**演讲稿**：
> CoT 解决了"思考"的问题。但它有一个根本局限——
>
> 模型的推理完全建立在它自己的内部知识上。
> 如果它不知道某个事实，比如"当前生产环境的配置文件里 server.port 是多少"，
> 就算让它 step by step 想一百步，它也不可能凭空知道。
>
> CoT 赋予了 LLM 推理的能力，但它仍然没有"手脚"——
> 它只能在自己的脑子里想，不能去外部世界验证。
>
> 这个问题在 2022 年 10 月被一篇论文解决了——ReAct。
> 它的思路非常直接：如果 CoT 让模型会思考，那把"思考"和"行动"交替进行呢？
> 每一步思考之后去执行一个操作，看到结果再思考下一步。

**伪代码/图表**：
```
CoT 能做:                           CoT 不能做:
  "3×2=6, 2×3=6, 6+6=12"          "当前线上 server.port 是多少？"
  "这个 bug 可能在第 42 行"          "把第 42 行的代码改一下"
  "根据概率论，答案是..."           "跑一下 test 看看改对了没有"

CoT ∈ Agent 的 Think 环节
但 Agent 还需要 Act → 这就是 ReAct 要解决的问题
```

## 三、ReAct：思考 + 行动（5-6 页）

----

### Slide 8：ReAct —— 思考之后，付诸行动

**核心要点**：
- ReAct 论文（arXiv 2022.10.06），Google Research
- 论文引用了 CoT：CoT 证明 LLM 可以通过"写出中间步骤"来提高推理准确率
- ReAct 的推广：如果中间步骤不只是文字推理，还可以包含外部行动呢？
- Thought 不一定是 CoT 式的分步推理——它泛指 LLM 对当前状态的判断和下一步的决策

**演讲稿**：
> 2022 年 10 月——ChatGPT 发布前不到两个月——Google Research 发表了 ReAct。
>
> 这篇论文在开头就引用了 CoT。CoT 证明了一件事：让 LLM 把中间推理过程写出来，
> 推理准确率显著提高。
>
> ReAct 的推广思路是：如果"中间步骤"不只是文字推理，还可以是**对外部世界的操作**呢？
>
> 每一步：LLM 先输出一段 Thought——对当前状态的判断和下一步的决策。
> 然后执行一个 Action——调用外部工具。工具返回的结果就是 Observation。
> LLM 拿这个 Observation 重新思考，再决定下一步做什么。
>
> 这里的 Thought 不一定是 CoT 那种"第 1 步 xxx，第 2 步 xxx"的形式化推理。
> 它可以很简单——"信息够了，直接回答"——也可以很复杂——分析 Observation 后判断需要换个关键词重新搜索。
> Thought 的本质是这场"推理+行动"交替运行中的**决策环节**，而不是特定的输出格式。
>
> 而 Thought → Action → Observation → Thought → ... 这个循环，
> 就是今天所有 Agent 的骨架——Agent Loop。

**伪代码/图表**：
```
CoT:                             ReAct:
  LLM 输出推理步骤                  LLM 输出 Thought + Action
        │                                  │
        ▼                                  ▼
  "第1步...第2步...第3步..."        Thought: "我需要搜索这个主题"
  纯文字推理，不接触外部世界          Action: search("Python Agent")
                                    Observation: "搜索结果: ..."
                                          │
                                          ▼
                                    Thought: "结果不理想，换关键词重搜"
                                    Action: search("Python Agent 2024")
                                    Observation: "搜索结果: ..."
                                          │
                                          ▼
                                    Thought: "现在信息足够了，可以回答"

CoT 在头脑里一步步想              ReAct 每想一步做一步，观察结果再想下一步
```

### Slide 9：ReAct 论文的实际实现 —— 硬编码 Tool Use

**核心要点**：
- ReAct 论文自带可运行原型，它的 Action 是硬编码实现的
- LLM 输出 Action 文本（如 "Search[Inception director]"）
- 外部代码用正则解析 → 调 Wikipedia API → 结果作为 Observation 拼回 Prompt
- 这是 Agent Loop 的第一个工业实现

**演讲稿**：
> ReAct 不是一篇纯理论论文——它自带了一个可运行的原型。
> 这个原型的 Action 是怎么实现的？说出来大家可能会觉得眼熟：
>
> LLM 输出 "Action: Search[Inception director]"
> 外部代码用正则表达式提取出 Search 和参数
> 去调 Wikipedia API
> API 的返回值就是 Observation，拼回 Prompt 继续
>
> 这就是最早的 Tool Use 方式——硬编码解析。
> 非常简单，但也能看出来非常脆弱——一旦 LLM 输出的格式稍有偏差，
> 正则匹配失败，整个 ReAct 循环就断了。
>
> 但这个原型证明了：Thought → Action → Observation 的循环是可行的。
> Agent Loop 的骨架，在 2022 年 10 月就已经确立了。

**伪代码/图表**：
```python
# ReAct 论文中 HotpotQA 任务的原始实现（简化伪代码）

messages = [{"role": "system", "content": "你是Agent，可以调用 Search[...] 工具"}]

for step in range(max_steps):
    response = call_llm(messages)

    # 用正则解析 LLM 输出的 "Search[关键词]"
    match = re.search(r'Search\[(.*?)\]', response)
    if match:

    else:
        return response  # 最终答案
```

### Slide 10：提取 Agent Loop —— ReAct 给我们的遗产

**核心要点**：
- Agent Loop = ReAct 范式的工程抽象
- 三行伪代码解释整个 Agent 的工作原理
- 这不是某个框架的发明，是 ReAct 论文确立的模式

**演讲稿**：
> 从 ReAct 的实现中，我们可以提取出一个通用的 Agent Loop 结构。
> 这个结构在之后的每个阶段都会反复出现——
> Function Calling 时代用它、MCP 时代用它、今天的 Claude Code 仍然用它。
>
> 之所以叫 Loop，因为它的本质就是一个 while 循环：
> 只要任务没完成，就不断让 LLM 根据当前观察决定下一步。
> 如果 LLM 判定完成了，就退出循环，返回最终答案。
>
> 这个循环的优雅之处在于——它把 AI Agent 的所有复杂度
> 浓缩成了三个问题：
> 1. Think 环节怎么做得更好？（CoT 演进）
> 2. Act 环节怎么做得更可靠？（工具调用演进——我们下一章的内容）
> 3. 上下文会不会爆？（记忆管理——再下一章）

**伪代码/图表**：
```python
# Agent Loop —— 这个结构在之后每个阶段都不会变
# 变的只是 Act 环节的实现质量

def agent_loop(task, max_steps=10):
    messages = [{"role": "system", "content": "你是一只猫娘..."}]
    observation = ""

    for step in range(max_steps):
        # THINK: LLM 根据 (任务 + messages历史 + 当前观察) 做推理
        prompt = messages + [{"role": "user", "content": observation}]
        response = call_llm(prompt)

        thought, action, action_input = parse(response)

        if action is None:           # LLM 判定任务完成
            return thought           # ← 循环出口

        # ACT: 执行工具
        result = execute_tool(action, action_input)

        # OBSERVE: 结果成为下一轮 THINK 的输入
        messages.append({"role": "assistant", "content": response})
        messages.append({"role": "user", "content": f"Observation: {result}"})
        observation = result
```

```
       ┌──────────┐    ┌──────────┐    ┌──────────┐
       │ Observe  │───→│  Think   │───→│   Act    │
       │ (观察)   │    │ (思考)   │    │ (行动)   │
       └──────────┘    └──────────┘    └──────────┘
            ↑                               │
            └───────────────────────────────┘
```

### Slide 11：Agent Loop 运行时浮现的两个工程问题

**核心要点**：
- 问题一（容量）：messages 每轮都在增长 → 迟早超出 Context Window
- 问题二（注意力）：上下文越长，LLM 越容易"忘记" System Prompt 中的格式指令
- 这两个问题一纵一横，共同决定了 Agent 能跑多远

**演讲稿**：
> 好，我们有了 Agent Loop，它能跑，能完成任务。但如果你把这个循环跑起来，
> 很快就会撞到两个工程问题。
>
> 第一个问题很直观——我们每轮都在 messages 里塞东西。
> Thought 一段，Action 一段，Observation 一段。
> 跑 5 步上下文可能就两三千字出去了，跑 10 步就是五六千字。
> 而模型的 Context Window 不是无限的。超出去了怎么办？截断？压缩？
>
> 第二个问题更隐蔽——我们刚才在 System Prompt 里写了"你必须输出 JSON 格式"。
> 当上下文只有几百字时，模型注意力集中，能稳定遵守。
> 但当上下文长到几千字、几万字，这条指令在整体中的权重被稀释了。
> 模型开始"遗忘"它——输出自然语言而不是 JSON → 解析失败 → 循环中断。
>
> 下一个 Slide 我们用同一段代码的两次不同运行，来直观展示这个效果。

**伪代码/图表**：
```
每轮循环 messages 的膨胀:

Step 1: [System, User, Assistant+Action, User+Observation]               ~400 tokens
Step 3: [System, ...(3轮历史)...]                                        ~1,200 tokens
Step 5: [System, ...(5轮历史)...]                                        ~2,000 tokens
Step 10: [System, ...(10轮历史)...]                                      ~4,000 tokens

问题一: messages 迟早撞上 Context Window 天花板 → 截断/压缩/丢弃？
问题二: System Prompt 在总上下文中的注意力占比持续下降
        Step 1: ~40%  Step 3: ~15%  Step 5+: <8%  → 格式指令被稀释
```

### Slide 12：注意力漂移——Prompt JSON vs 原生 Function Calling

**核心要点**：
- Prompt 强制 JSON —— 格式约束在文字里，上下文越长越容易失效
- 原生 Function Calling —— 格式约束在 API 的 tools 参数里，不受上下文影响
- 同一任务对比：Prompt JSON Step 4 崩溃，原生 FC Step 8 仍然稳定

**演讲稿**：
> 我用同一段任务跑了两种模式。
>
> 左边是 Prompt 强制 JSON——在 System Prompt 里写"你必须输出 JSON"。
> 前 3 步正常，第 4 步模型开始输出自然语言而不是 JSON——解析崩溃，循环中断。
>
> 右边是原生 Function Calling——工具 Schema 放在 API 的 tools 参数里，
> 不在 Prompt 里写格式要求。跑到第 8 步仍然稳定。
>
> 核心区别在哪？
> Prompt JSON 的约束是"文字级"的——LLM 需要在每轮推理时"记住" JSON 格式，
> 而上下文越长这个"记忆"越不可靠。
> 原生 FC 的约束是"API 级"的——工具 Schema 由 API 层保证，
> LLM 正常输出自然语言，想调工具时 API 在 tool_calls 字段返回结构化调用。
>
> 更麻烦的是，Prompt JSON 的漂移不是稳定复现的——同样的输入，
> 有时 Step 4 漂移，有时 Step 5 漂移。测试过了生产崩了，回来复现又过了。
> 这是所有工程师最头疼的间歇性故障。

**伪代码/图表**：
```python
# ❌ Prompt 强制 JSON — 约束在 Prompt 文字里，随上下文衰减
SYSTEM = """你必须输出 JSON: {"thought": "...", "action": "...", ...}
可用工具: search_files, read_file, run_tests"""
#     ↑ 每轮 LLM 都要"记住"这个格式要求。上下文越长，越容易忘。

# ✅ 原生 Function Calling — 约束在 API 的 tools 参数里
SYSTEM = "你可以使用工具来搜索文件、读取内容、运行测试。"
TOOLS = [{"type": "function", "function": {"name": "search_files", ...}}]
response = call_llm(messages, tools=TOOLS)
#      ↑ API 保证 tool_calls 结构，不受上下文长度影响
```

```
实际对比（同一任务，真实运行数据）:

📦 Prompt 强制 JSON:              🌐 原生 Function Calling:
  Step 1 ✅ JSON output             Step 1 ✅ tool_calls
  Step 2 ✅ JSON output             Step 2 ✅ tool_calls
  Step 3 ✅ JSON output             Step 3 ✅ tool_calls
  Step 4 ❌ 漂移！                  Step 4 ✅ tool_calls
  parse() 失败 → 退出               Step 5 ✅ tool_calls
                                    Step 6 ✅ tool_calls
                                    Step 7 ✅ tool_calls
                                    Step 8 ✅ 正常结束

原因: 约束靠 Prompt 文字维持       原因: 约束靠 API tools 参数保证
     上下文越长约束越弱                  不受上下文长度影响
     间歇性失效 → 难调试                 确定性工作 → 可靠
```

### Slide 13：Agent Loop 的第三个问题——记忆

**核心要点**：
- 短期记忆（messages）容量有限，长期记忆（RAG/DB）容量近乎无限
- 百万上下文 ≠ 好记忆：Lost in the Middle + 容量 ≠ 检索能力
- 好的 Memory 设计是在"有限的 Context 里放最有价值的信息"

**演讲稿**：
> 还有一个问题——记忆。
>
> Agent 每次重启 messages 都是空的。上次任务中学到的经验全丢了。
> 就算在同一个会话里，messages 也不可能无限增长。
>
> 所以 Agent 的记忆需要分层：
> 短期记忆就是 messages 数组——当前会话的工作区，容量受 Context Window 限制。
> 长期记忆存在外部——向量数据库、传统数据库、文件系统——跨会话可用。
>
> 这里要澄清一个常见误解。
> 厂商说"支持 100 万 Context"，很多人以为 Agent 就有了好记性。
> 但两个原因让这不是一回事。
> 第一，注意力不均匀。模型对开头和结尾记得清楚，
> 中间的部分容易被稀释——叫 Lost in the Middle。
> 第二，容量不等于检索能力。给你一本大书和给你一本大书加 Ctrl+F，效率天差地别。
>
> 所以 Memory 的本质不是"模型能塞多少"，而是
> "你怎么在有限 Context 里放最有价值的信息"。
> 这个问题我们后面讲上下文压缩和渐进式披露时会展开。

**伪代码/图表**：
```
Agent 的三层记忆:

│ 短期记忆   │ messages 数组              │ Context Window │ 当前会话 │
│ 工作记忆   │ 变量/Plan/当前Observation   │ 极有限          │ 当前步骤 │
│ 长期记忆   │ RAG/向量库/DB/文件系统      │ 近乎无限        │ 跨会话   │

记忆流转:
  Observation → 加入 messages（短期）
              ↓
  messages 逼近 Context 上限 → 触发压缩
              ↓
  关键信息存入长期记忆 → 短期用摘要替换旧 messages
```

```
上下文注意力分布（Lost in the Middle）:
  [System Prompt] ...大量历史消息... [最新消息]
  █████░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░████
  开头清晰             中间模糊           结尾清晰
```

### Slide 14：回顾——ReAct 的 Act 是如何实现的

**核心要点**：
- ReAct 确立了 Think→Act→Observe 循环
- 但 Act 环节的实现一直很原始：正则解析 LLM 输出的文本
- 从 2023 到 2025，工业界在解决一个问题：Act 如何变得可靠

**演讲稿**：
> 好，我们现在有了 Agent Loop。但回忆一下 Slide 9——ReAct 论文里 Act 是怎么实现的？
>
> LLM 输出 "Search[Inception director]"，外面用正则表达式抠出 Search 和参数，去调 Wikipedia API。
>
> 这能跑。但正则解析天然脆弱：格式稍有偏差就崩。
> 而且每加一个新工具，就得写一套新的解析逻辑。
>
> 所以从 Function Calling（2023.06）到 MCP（2024.11）到 Agent Skills（2025.10），
> 整整两年半，工业界在做一个看似无聊但至关重要的事——
> **把 Act 环节从"能用"升级到"可靠"**。

**伪代码/图表**：
```
ReAct 的原始 Act 实现:

  LLM 输出: "Action: Search[Inception director]"
            ↓
  正则解析: re.search(r'Search\[(.*?)\]', output)
            ↓
  执行工具: call_wikipedia_api("Inception director")
            ↓
  拼回结果: messages.append({"role":"user","content":result})
            ↓
  下一轮:   LLM 继续推理

痛点:
  1. LLM 输出格式不固定 → 正则匹配概率性失败 ← Slide 12 的注意力漂移
  2. 新工具 → 新正则 → 维护噩梦
  3. 所有工具挤在同一个进程里 → 没有隔离、不可复用
```

### Slide 15：Function Calling —— 结构化的工具调用

**核心要点**：
- OpenAI 2023.06.13 推出 Function Calling API
- 核心变化：工具 Schema 放在 API 的 `tools` 参数里，不在 Prompt 里
- LLM 想调工具 → API 在 `tool_calls` 字段返回结构化调用
- 彻底解决了正则解析的脆弱性

**演讲稿**：
> 2023 年 6 月，OpenAI 发布了 Function Calling API。
>
> 它的核心变化只有一点——工具的定义从 Prompt 里搬到了 API 参数里。
>
> 以前在 Prompt 里写"可用工具: search_files(query: str)"，模型可能忘、可能写错格式。
> 现在把工具 Schema 放到 API 的 tools 参数里，模型决定调工具时，
> API 直接在 tool_calls 字段里返回 `{"name": "search_files", "arguments": {"query": "xxx"}}`。
>
> 你不用再写正则了。格式由 API 保证。
>
> 这是 Act 环节的第一个质变：从"大概率能解析"变成了"确定能解析"。

**伪代码/图表**：
```python
# ❌ 之前：正则解析，在 Prompt 里描述工具
SYSTEM = "可用工具: search_files(query), read_file(path)"
response = call_llm(messages)
match = re.search(r'TOOL:\s*search_files\("(.+?)"\)', response)

# ✅ 之后：结构化 Function Calling，Schema 在 API 参数里
TOOLS = [{
    "type": "function",
    "function": {
        "name": "search_files",
        "description": "搜索文件",
        "parameters": {
            "type": "object",
            "properties": {"query": {"type": "string"}},
            "required": ["query"]
        }
    }
}]
response = call_llm(messages, tools=TOOLS)
# API 返回:
# {"choices":[{"message":{"tool_calls":[{
#     "function":{"name":"search_files","arguments":"{\"query\":\"test\"}"}
# }]},"finish_reason":"tool_calls"}]}
#                     ↑ 格式由 API 层保证，不需要正则
```

### Slide 16：Function Calling 之后的新问题 → MCP

**核心要点**：
- FC 解决了格式，但没解决"工具从哪来"和"怎么管理"
- 工具仍要手动注册在代码里，换一个 Agent 要重新配置
- 不同工具服务的接口不统一，无法跨进程调用
- MCP（2024.11）提出：用一个标准协议连接 Agent 和工具

**演讲稿**：
> Function Calling 让 Act 环节的格式问题解决了。但很快有了新问题。
>
> 工具的 Schema 还是在代码里硬编码的。比如你要接一个数据库、一个文件系统、一个 Slack——
> 每个工具的 Schema 和实现都要手动写，然后注册到 Agent 里。
>
> 如果你有十个 Agent，每个 Agent 要用不同的工具组合，怎么办？
> 如果工具运行在另一个进程、另一台服务器上，怎么办？
>
> 2024 年 11 月，Anthropic 提出了 MCP（Model Context Protocol）。
> 它的思路和 Function Calling 有本质区别——
> FC 解决的是"工具调用的格式"，MCP 解决的是"工具如何被发现和连接"。

**伪代码/图表**：
```
Function Calling 解决的问题:            MCP 解决的问题:

  Prompt 里写工具描述                    Agent 启动时自动发现工具
  → FC 把 Schema 放进 API 参数           → 调 tools/list 拿到可用工具列表
  → 返回结构化的 tool_calls              → 不需要在代码里硬编码工具

                                        工具运行在独立进程/服务器
                                        → Client-Server 架构
                                        → stdio / HTTP 通信
                                        → 工具崩溃不影响 Agent
```

----

### Slide 17：MCP 架构 —— 工具发现与标准化通信

**核心要点**：
- MCP 是 Client-Server 架构：Agent 是 Client，工具服务是 Server
- `tools/list`：Agent 动态发现 Server 提供哪些工具
- `tools/call`：Agent 调用工具，Server 返回结果
- 传输层无关：stdio（本地进程）或 HTTP/SSE（远程服务）

**演讲稿**：
> MCP 把 Agent 和工具的关系变成了 Client 和 Server。
>
> Agent 启动时，不是去读代码里的工具列表，而是向 Server 发一个 tools/list 请求。
> Server 返回它提供的所有工具的 Schema。
> Agent 需要调工具时，发 tools/call，Server 执行并返回结果。
>
> 这意味着什么？工具可以从 Agent 代码中彻底解耦出来。
> 你可以用本地进程通信（stdio），也可以走 HTTP 连远程服务。
> 同一个 MCP Server 可以被多个 Agent 复用，不需要每个 Agent 单独写一次工具注册代码。
>
> 这是 Act 环节的第二个质变：从"代码耦合"变成了"协议解耦"。

**伪代码/图表**：
```
MCP 协议流程:

  ┌─────────────┐                         ┌──────────────────┐
  │  Agent       │     stdio / HTTP        │  MCP Server      │
  │  (Client)    │←───────────────────────→│  (Tool Host)    │
  │             │                         │                  │
  │  1. tools/list:              →        │  返回:           │
  │     "你有哪些工具？"                   │  [{name:"read_file",
  │                                       │    params:{...}},
  │  2. tools/call:              →        │   {name:"query_db",
  │     name:"read_file"                   │    params:{...}}]
  │     arguments:{path:"/app/main.py"}    │
  │                                       │  执行 read_file
  │                      ← 返回结果         │  返回文件内容
  └─────────────┘                         └──────────────────┘
```

```
核心方法:

  tools/list       → Agent 动态发现工具（不需要在代码里硬编码）
  tools/call       → Agent 调用工具（统一接口）
  resources/list   → 列出 Server 暴露的数据资源
  resources/read   → 读取指定资源

MCP 的进化意义:
  Act 环节从 "Agent 代码中硬编码工具" → "Agent 通过协议动态发现和调用工具"
  工具可独立部署、独立版本化、跨 Agent 复用
```

### Slide 18：Agent Skills —— 工具的"工程封装"

**核心要点**：
- Skill = Prompt Template + Tools + Domain Knowledge + Workflow
- 把相关工具、知识、流程打包为一个可复用的"技能"
- 2025.10 Anthropic 发布，同年 12 月成为开放标准（agentskills.io）

**演讲稿**：
> MCP 解决了工具如何被 Agent 发现和调用。但还有一个问题没解决——
>
> 一个"代码审查"功能，不只是调用 lint 工具就完了。
> 你需要先读文件 → 跑静态分析 → 检查常见漏洞模式 → 生成审查报告。
> 这些步骤 + 工具 + 审查知识，是一个整体。分开管理太零碎。
>
> Agent Skills 的思路是：把 Prompt + 工具 + 领域知识 + 执行流程打包成一个 Skill。
> Agent 需要时加载这个 Skill——它带着自己的指令、自己的工具、自己的知识进来。
>
> 比如一个"代码审查 Skill"，里面已经内置了常见安全漏洞模式，
> Agent 不用每次都从头学 SQL 注入怎么检查。
>
> 这是 Act 环节的第三个质变：从"调单个工具"变成了"加载一组能力"。

**伪代码/图表**：
```
Skill 的结构:

  ┌───────────────────────────────────────┐
  │  Skill: "CodeReview"                  │
  │                                       │
  │  Prompt Template: "请审查以下代码..."   │
  │  Tools: [read_file, run_linter]       │
  │  Domain Knowledge:                    │
  │    - SQL 注入模式识别                 │
  │    - XSS 漏洞模式识别                 │
  │    - 敏感信息泄露检查                  │
  │  Workflow:                            │
  │    Step 1: 读取目标文件               │
  │    Step 2: 运行静态分析               │
  │    Step 3: 基于 Knowledge 检查漏洞     │
  │    Step 4: 生成审查报告               │
  └───────────────────────────────────────┘

Skill vs 硬编码工具:

            │ 硬编码工具         │ Agent Skills               │
  维护成本   │ 散落在代码各处     │ Skill 内聚，独立维护       │
  复用性     │ 工具级，手动组合   │ Skill 级，开箱即用         │
  上下文     │ Schema 恒占 Token  │ 按需加载，不用不占         │
  知识     │ 仅工具描述          │ 含领域知识和最佳实践       │
```

### Slide 19：Act 进化总结 —— 一条线串起三年

**核心要点**：
- Act 环节在三年内经历了三次质变
- 每一次都解决了上一阶段的核心痛点
- Think→Act→Observe 循环不变，但 Act 的质量持续提升

**演讲稿**：
> 我们把 Act 环节的进化串起来看。
>
> 2022 年的 ReAct，Act 是用正则解析的。能用，但脆弱。
> 2023 年的 Function Calling，把格式约束从 Prompt 搬到了 API 层。格式再也不丢了。
> 2024 年的 MCP，把工具从 Agent 代码中解耦，变成可发现、可复用的服务。
> 2025 年的 Agent Skills，把工具、知识和流程打包成可加载的能力单元。
>
> 三年时间，Act 环节从一行 `re.search` 变成了一个标准化的协议生态。
> 而整个过程中，Agent Loop 的 Think→Act→Observe 循环结构从来没变过。
> 变的只是 Act 环节越来越可靠。
>
> 但 Act 可靠了，新的问题又来了——Agent 的"眼界"问题。这就是下一节 Plan-and-Execute。

**伪代码/图表**：
```
Act 环节进化:

  2022.10  ReAct 原始实现        — 正则解析 LLM 输出
  2023.06  Function Calling      — 结构化工具调用，格式由 API 保证
  2024.11  MCP                   — 标准化协议，工具发现与通信解耦
  2025.10  Agent Skills          — 工具+知识+流程 的能力封装

  不变: Think → Act → Observe 循环结构
  变化: Act 从 "脆弱" → "可靠" → "可扩展" → "可复用"

Act 可靠之后 → 新问题浮现:
  Agent 每步"走一步看一步"（ReAct），复杂任务可能偏离方向
  → 能不能在 Loop 之上加一层全局规划？
  → Plan-and-Execute
```

## 五、Plan-and-Execute：从"走一步看一步"到"先规划再执行"（2-3 页）

----

### Slide 20：ReAct 的局限——会走，但不会看路

**核心要点**：
- ReAct 每一步决策只基于当前 Observation + 历史
- 没有对整个任务的全局规划 → 可能在中间步骤迷路
- 类比：ReAct 是导航的"下一步左转"，Plan-and-Execute 是"先看全程路线"

**演讲稿**：
> Act 环节现在已经很可靠了——工具调用格式稳定，通信协议标准化，能力可以封装复用。
> 但 Agent 还有一个更上层的问题。
>
> ReAct 的决策是"走一步看一步"——
> 每一步的 Thought 只基于当前 Observation 和历史记录。
> 它没有一个全局的"路线图"。
>
> 打个比方：ReAct 像一个只有下一步导航的司机——"200 米后左转"。
> 短途还行，长途很容易走偏。
> Plan-and-Execute 是先看全程地图，把路线分解成段，然后一段段走。
>
> 这个想法由 2023 年的一篇论文 Plan-and-Solve Prompting 正式提出，
> 后来在工程中演化为 Planner + Executor 的两阶段架构。

**伪代码/图表**：
```
ReAct:                              Plan-and-Execute:

  Thought → Action → Obs             Planner: 把任务拆成子任务
  Thought → Action → Obs                  │
  Thought → Action → Obs             [Task1, Task2, Task3, Task4]
  ...                                     │
  每步决策只看到"当前这一步"         Executor: 用 ReAct Loop 执行每个子任务
                                            │
                                     Task1 → Thought→Action→Obs
                                     Task2 → Thought→Action→Obs
                                     ...

ReAct 是微观决策                   Plan-and-Execute 是宏观规划 + 微观执行
```

----

### Slide 21：Planner + Executor 两阶段架构

**核心要点**：
- Planner：将复杂任务分解为有序的子任务列表
- Executor：使用 ReAct Loop 执行每个子任务
- 动态重规划：某个子任务失败 → Planner 调整后续计划

**演讲稿**：
> 这个架构分两层。
>
> Planner 的任务是把用户的大任务拆成小任务。
> "部署这个应用到服务器" → 
> 1. 检查 Dockerfile，2. 检查依赖，3. 构建镜像，4. 运行测试，5. 部署。
>
> Executor 的任务是用我们熟悉的 ReAct Loop 去执行每个子任务。
> 每完成一个，把结果汇报给 Planner。如果某个子任务失败了，
> Planner 可以根据失败原因重新规划后续步骤——这叫动态重规划。
>
> 现代 Agent 系统基本上是 Plan-and-Execute 和 ReAct 的混合：
> Planner 负责"想清楚做什么"，Executor 负责"把事做到位"。

**伪代码/图表**：
```python
# Plan-and-Execute 核心伪代码

def plan_and_execute(task):
    # Phase 1: Plan
    plan = planner.create_plan(task)
    # plan = {
    #   "subtasks": [
    #     {"id":1, "title":"检查Dockerfile", "tools":["read_file"]},
    #     {"id":2, "title":"检查依赖", "tools":["read_file","search"]},
    #     {"id":3, "title":"构建镜像", "tools":["run_command"]},
    #   ]
    # }

    # Phase 2: Execute
    for subtask in plan["subtasks"]:
        result = executor.execute(subtask)   # ← 内部用 ReAct Loop
        if result.status == "failed":
            plan = planner.revise(plan, result)  # ← 动态重规划
```

```
Planner + Executor 协作:

           用户任务: "部署应用到服务器"
                    │
        ┌───────────┴───────────┐
        ▼                       ▼
  ┌──────────┐           ┌──────────┐
  │ Planner  │──────────→│ Executor │
  │ 任务分解  │  子任务列表 │ 逐步执行  │
  └──────────┘           └──────────┘
        ↑                       │
        │                       │ 子任务失败
        │                       ▼
        └──── 动态重规划 ←──────┘
```

### Slide 22：单 Agent 的瓶颈 → Multi-Agent

**核心要点**：
- Plan-and-Execute 解决了单个 Agent 的规划问题
- 但单个 Agent 面对多领域复杂任务仍然吃力
- 自然的思路：让多个专业 Agent 协作

**演讲稿**：
> Plan-and-Execute 让单个 Agent 能处理复杂任务了。
> 但它终究是一个 Agent——一个大脑。
>
> 现实中的复杂项目不是一个人能完成的——需要开发者、测试、架构师、运维协作。
> 那 AI Agent 为什么不能也这样？
>
> 这就是 Multi-Agent 的思路：不是让一个 Agent 变全知全能，
> 而是让多个专业 Agent 各自负责自己擅长的领域，协作完成一个任务。
>
> 这个方向现在还在快速演进中，我们看几个已经成型的协作模式。

**伪代码/图表**：
```
单 Agent:                           Multi-Agent:

  ┌──────────┐                     ┌──────┐  ┌──────┐  ┌──────┐
  │ Agent    │                     │Coder │  │Review│  │Test  │
  │ (全栈)    │                     │ Agent│→│ Agent│→│ Agent│
  └──────────┘                     └──────┘  └──────┘  └──────┘
  能力分散，容易出错                  各司其职，交叉验证
```

## 六、Multi-Agent 与前沿（2-3 页）

### Slide 23：Multi-Agent 协作模式

**核心要点**：
- Sequential：流水线，前一个的输出是后一个的输入
- Hierarchical：Master Agent 调度 Worker Agent
- Debate：多 Agent 讨论达成共识

**演讲稿**：
> Multi-Agent 协作目前有三个主要的工程模式。
>
> Sequential 最简单——Coder 写完 → Reviewer 审查 → Tester 测试。
> 像工厂流水线，每个 Agent 专注一个环节。
>
> Hierarchical 最灵活——一个 Master Agent 当"项目经理"，
> 根据任务类型分派给不同的 Worker。写代码的找 Coder，做分析的找 Analyst。
>
> Debate 最有意思——两个 Agent 各自给出分析和理由，
> 通过讨论消除单一 Agent 的偏见和幻觉。
>
> 这三种模式可以组合使用。现在生产环境中用得最多的是 Sequential + Hierarchical 混合。

**伪代码/图表**：
```
Sequential:
  [Agent A] → [Agent B] → [Agent C] → Result

Hierarchical:
       [Master Agent]
        /    |    \
  [Agent A][Agent B][Agent C]
        \    |    /
       [Merge Result]

Debate:
  [Agent A] ←→ [Agent B]
       ↘      ↙
     [Consensus]
```

### Slide 24：Computer Use 与 Code Agent

**核心要点**：
- Computer Use：Agent 操作 GUI 界面——看图、点击、输入
- Code Agent：目前最成熟的 Agent 应用——读写代码、执行命令、Debug
- 共同趋势：从"调用 API"扩展到"操作真实世界"

**演讲稿**：
> 最后说两个正在发生的前沿方向。
>
> Computer Use 让 Agent 能像人一样操作计算机——
> 看屏幕截图、点鼠标、敲键盘。这意味着即使没有 API 的传统软件，
> Agent 也能操控。这个方向还比较早期，但想象空间很大。
>
> Code Agent 是目前最成熟的 Agent 应用——
> Claude Code、Cursor、Copilot 都是这方面的代表。
> 它们的核心就是 Agent Loop + 文件系统工具 + 终端工具 + 代码智能。
> 读取、修改、执行、测试、Debug——完整闭环。
>
> 这两个方向的共同点：Agent 的能力边界在从"调用 API"扩展到"操作真实世界"。

**伪代码/图表**：
```
Computer Use:                          Code Agent (今天最成熟的 Agent):

  See: 截图 → 识别UI元素                Read: 读代码、读配置、读错误日志
  Think: 决定操作什么                   Think: 分析问题、设计修改方案
  Act: 点击/输入/滚动                   Act: 写文件、执行命令、运行测试
  Observe: 看屏幕变化                   Observe: 看输出、看报错、看测试结果
```

### Slide 25：全篇总结——一条线串起六年

**核心要点**：
- 从 Chat 到 Multi-Agent，核心驱动力始终是：如何让 LLM 更可靠地"做事"
- Agent Loop（Think→Act→Observe）是贯穿始终的不变骨架
- 每个阶段的创新都在增强这个循环的某一环

**演讲稿**：
> 我们用最后一页来串一下整个故事。
>
> 2020-2022，LLM 从 Completion API 走到 ChatGPT——模型会聊天了。
> 2022.01，CoT 让模型学会"一步步思考"。
> 2022.10，ReAct 把思考和行动结合——Agent Loop 的骨架诞生。
> 但当时的 Act 环节极其脆弱——正则解析 LLM 的文本，格式随时可能漂移。
>
> 2023 年，Function Calling 解决了格式问题——约束从 Prompt 文字搬到了 API 层。
> 2024 年，MCP 解决了通信问题——工具从 Agent 代码中解耦，变成标准化的 Server。
> 2025 年，Agent Skills 解决了工程问题——工具、知识、流程封装为可复用单元。
>
> 工具可靠了，Plan-and-Execute 给 Agent 加上了全局规划能力。
> 单个 Agent 不够用了，Multi-Agent 让多个 Agent 协作。
> 今天，Computer Use 和 Code Agent 在把 Agent 的能力从"调 API"扩展到"操作真实世界"。
>
> 整个故事的暗线只有一条：
> Agent 在变得**越来越能做事**，而且**越来越可靠地在做事**。

**伪代码/图表**：
```
2020 ───── 2022 ───── 2023 ───── 2024 ───── 2025 ───── 2026

Completion  ChatGPT   Function   MCP        Agent      Multi-Agent
  API      发布      Calling    协议       Skills     协作
   │         │          │         │          │           │
   ▼         ▼          ▼         ▼          ▼           ▼
 能补全    能聊天     能做判断   能发现工具  能加载能力  能团队协作

贯穿始终: Think → Act → Observe
           思考    行动    观察 → 从正则到协议，始终在增强 Act 的可靠性
```
